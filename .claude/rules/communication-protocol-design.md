# Communication Protocol

## 適用範圍

所有團隊內部溝通與跨團隊溝通（與 Web-Dev-Team）都必須遵守此協議。

---

## 溝通時機

### 與 Web-Dev-Team 的溝通時機（必須主動通知）

**里程碑完成時**：
- 客戶第 1 輪確認完成 → 通知進度
- 客戶第 2 輪最終簽核 → 通知即將交接
- 程式碼完成 QA 驗收 → 通知準備交接

**發現問題時**（即時通知）：
- 設計變更影響後端 API 結構
- 發現客戶需求與技術可行性衝突
- 預計交期延遲

**每週定期更新**（每週五）：
- 進行中專案的狀態更新
- 預計下週交接的專案清單

---

## 溝通管道

### 非緊急溝通（48 小時內回覆）
- **Email**：正式通知、文件傳送
- **Slack #web-design-handoff**：進度更新、一般問題

### 緊急溝通（4 小時內回覆）
- **Slack DM**：緊急問題、阻礙事項
- **電話/會議**：重大技術決策、客戶緊急變更

---

## 溝通格式範本

### 進度更新範本
```
【進度更新】{專案名稱}

目前階段：階段 X - {階段名稱}
完成度：X%
預計交接日：{YYYY-MM-DD}
狀態：正常 / 延遲 / 阻礙

阻礙事項（若有）：
- {描述}
```

### 問題通報範本
```
【問題通報】{專案名稱}

問題類型：設計變更 / 技術問題 / 客戶需求變更
影響範圍：前端 / 後端 / 資料庫
緊急程度：高 / 中 / 低

問題描述：
- {詳細描述}

建議解決方案：
- {若有}
```

### 交接通知範本
```
【交接通知】{專案名稱}

交接日期：{YYYY-MM-DD}
交接會議時間：{時間}
會議連結：{若為線上會議}

交付物清單：
- {列出所有交付物}

請確認：
- [ ] 已收到所有檔案
- [ ] 已安排參加交接會議
```

---

## 團隊內部溝通規範

### 每日 Standup（非強制，依專案需要）
- 時間：每日早上 10:00
- 時長：15 分鐘
- 格式：昨日完成 / 今日計畫 / 阻礙事項

### 審核結果通知
- **通過**：直接在工作流程中標註，無需額外通知
- **未通過**：必須附上具體修改建議，並設定重新審核時間

---

## 違反判定

- 重大問題未即時通知 Web-Dev-Team → 違反
- 交接前未提前通知 → 違反
- 溝通未使用標準格式 → 警告（3 次警告 = 違反）
- 緊急溝通超過 4 小時未回覆 → 違反

---

## 例外

- 客戶機密專案可使用加密通訊管道
- 跨時區協作可調整回覆時間要求
